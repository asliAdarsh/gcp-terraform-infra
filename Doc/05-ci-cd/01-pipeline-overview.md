# CI/CD Pipeline Overview

**Folder:** `.github/workflows/`

All infrastructure changes go through GitHub Actions. No one runs `terraform apply` from their laptop after bootstrap.

## Workflows at a Glance

| Workflow File | Trigger | What It Does |
|---------------|---------|-------------|
| `core.yaml` | Push/PR to `Deployment/Core/**` or `Workload/Core/**` | Deploys team folder changes |
| `{team}.yaml` | Push/PR to `Deployment/Spoke/{Team}/**`, `Workload/Spoke/{Team}/**`, or `Modules/**` | Deploys team resources |
| `pr-check.yaml` | Any PR touching `Workload/**`, `Deployment/**`, or `Modules/**` | Format check + validation |
| `deploy-app.yaml` | Called by other workflows (reusable) | Shared apply/plan logic |

## Authentication Flow

```
GitHub Actions
  │  OIDC token (auto-generated by GitHub)
  ▼
Workload Identity Federation (GCP)
  │  Verifies token, maps to service account
  ▼
Stage Service Account (e.g., terraform-service-projects)
  │  Performs Terraform operations
  ▼
GCP Resources
```

No service account keys are stored in GitHub Secrets. Authentication uses GitHub's OIDC token, which is scoped to the repository.

## PR vs Push Behavior

| Event | Action |
|-------|--------|
| **Pull Request** | `terraform plan` only — see what would change |
| **Push to dev/staging/prod** | `terraform plan` + `terraform apply` |

This means you always see a plan in the PR before anything is applied.

## Environment Segregation

Each branch (dev, staging, prod) maps to a GCP environment. The `deploy-app.yaml` workflow passes the branch name as the environment variable, which determines which `.tfvars` file is used and which environment folder is targeted.
